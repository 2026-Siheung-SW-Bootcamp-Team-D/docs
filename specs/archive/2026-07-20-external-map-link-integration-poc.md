# 외부 지도 링크 왕복 연동 PoC 계획서

- 문서 버전: 0.1
- 작성일: 2026-07-20
- 대상 서비스: 여러 참여자의 모임 시간·장소 조율 B2C 웹 서비스
- 검증 대상: 카카오맵·네이버 지도 장소 링크 가져오기, 외부 지도 다시 열기, 상세 경로 역할 분담

## 1. PoC 목적

사용자가 카카오맵 또는 네이버 지도에서 찾은 장소 링크를 우리 서비스의 후보로 가져오고, 확정된 장소는 다시 사용자가 선호하는 지도 앱에서 열 수 있는지 검증한다.

이 PoC는 단순히 예제 URL 한 건을 파싱하는 데 목적이 없다. 실제 공유 링크의 변형, 단축 URL, 좌표 누락, 앱 미설치, 모바일·PC 차이와 보안 문제까지 확인해 MVP 지원 범위를 결정해야 한다.

검증 후 다음 질문에 답할 수 있어야 한다.

1. 어떤 카카오맵·네이버 지도 URL 형식을 자동으로 가져올 수 있는가?
2. 링크에 좌표가 없을 때 장소를 얼마나 정확하게 복원할 수 있는가?
3. 가져온 장소를 우리 지도 위에 정확히 표시할 수 있는가?
4. 확정 장소와 출발지를 외부 지도의 대중교통 길찾기로 넘길 수 있는가?
5. 우리 서비스는 어느 수준의 경로 정보를 남겨야 사용 가치가 유지되는가?
6. URL 처리 과정에서 임의 서버 접근, 잘못된 장소 등록, 외부 페이지 변경 위험을 통제할 수 있는가?

## 2. 제안하는 제품 경계

### 우리 서비스가 담당할 것

- 이동하기 편한 추천 지역 3개 제시
- 참여자가 등록한 정확한 장소 후보 관리
- 참여자별 예상 이동시간·환승·요금 비교
- 후보 투표와 최종 확정
- 확정 후 개인별 권장 출발시각과 간단한 경로 미리보기

### 외부 지도가 담당할 것

- 음식점·카페·놀거리 탐색과 리뷰 확인
- 실시간 배차와 최신 경로 재탐색
- 정류장별 상세 안내와 실제 이동 내비게이션

### 명시적 비범위

- 카카오맵·네이버 지도 페이지를 iframe으로 삽입
- 외부 지도 리뷰·사진·평점 스크래핑
- 모든 비공식 URL 형식의 영구 지원 보장
- 우리 서비스에서 턴바이턴 내비게이션 구현
- 외부 지도와 TMAP의 경로 결과가 완전히 같다고 보장

## 4. 수집할 테스트 URL

실제 사용자가 공유 버튼으로 복사한 URL을 중심으로 표본을 만든다. 문서에 있는 예제 URL만으로 합격시키지 않는다.

### 공급자별 URL 유형

| 공급자 | 유형 | 권장 표본 수 |
|---|---|---:|
| 카카오맵 | 좌표 포함 지도 URL | 5 |
| 카카오맵 | 장소 ID URL | 5 |
| 카카오맵 | 모바일 공유·단축 URL | 10 |
| 카카오맵 | 검색결과·길찾기 URL | 5 |
| 네이버 지도 | PC 장소 공유 URL | 10 |
| 네이버 지도 | 모바일 앱 공유·단축 URL | 10 |
| 네이버 지도 | 검색결과·길찾기 URL | 5 |
| 공통 | 깨진 URL·임의 도메인·악성 리다이렉트 | 10 이상 |

### 장소 유형

- 수도권 지하철역과 역사 내부 장소
- 유명 상권의 음식점·카페·복합문화공간
- 같은 상호가 여러 지점에 있는 프랜차이즈
- 건물명과 매장명이 다른 장소
- 지하철 없는 지방의 기차역·버스터미널·상업 거점
- 한글·영문·숫자·특수문자가 포함된 장소명
- 폐업·이전·검색 결과가 없는 장소

각 표본에는 다음 정답 데이터를 사람이 기록한다.

```json
{
  "provider": "KAKAO",
  "inputUrl": "사용자가 복사한 원본 URL",
  "expectedName": "장소명",
  "expectedAddress": "주소",
  "expectedLatitude": 37.0,
  "expectedLongitude": 127.0,
  "deviceSource": "ANDROID_APP",
  "urlType": "SHORT_SHARE"
}
```

API 키, 로그인 쿠키, 개인 위치가 포함된 URL은 표본 파일에 저장하지 않는다.

## 5. PoC 구성 요소

### 5.1 URL Classifier

입력 문자열이 URL인지 검사하고 공급자와 예상 형식을 분류한다.

출력 예시:

```json
{
  "provider": "NAVER",
  "urlType": "SHORT_SHARE",
  "requiresRedirectResolution": true
}
```

지원하지 않는 일반 웹 URL은 외부 요청 전에 거절한다.

### 5.2 Safe Redirect Resolver

단축 URL이 실제 지도 URL로 이동하는 경우에만 서버에서 리다이렉트를 따라간다.

필수 보호 규칙:

- `https`만 허용
- 카카오·네이버의 명시적 도메인 allowlist 적용
- 각 리다이렉트 단계마다 호스트 재검사
- 최대 리다이렉트 3회
- 연결·전체 timeout 설정
- 사설 IP, loopback, link-local 주소 차단
- 응답 본문 전체 다운로드 금지
- 최종 URL과 리다이렉트 횟수만 기록

### 5.3 Provider Parser

공급자별 파서가 URL에서 다음 값을 추출한다.

- 외부 장소 ID
- 장소명
- 주소 또는 검색어
- 위도·경도
- 출발지·목적지 여부
- 공급자와 원본 URL

카카오맵과 네이버 지도 파서는 독립 모듈로 만든다. 한 공급자의 URL 형식 변경이 다른 공급자나 도메인 모델에 영향을 주면 안 된다.

### 5.4 Place Normalizer

공급자별 결과를 내부 장소 형식으로 변환한다.

```json
{
  "sourceProvider": "KAKAO",
  "sourcePlaceId": "18577297",
  "sourceUrl": "https://...",
  "name": "카카오판교아지트",
  "address": "...",
  "latitude": 37.395296,
  "longitude": 127.110449,
  "resolutionMethod": "URL_COORDINATE",
  "verificationStatus": "CONFIRMED"
}
```

`sourcePlaceId`는 공급자 내부에서만 의미가 있다. 서로 다른 지도 공급자의 장소 ID를 같은 값으로 취급하지 않는다. 내부 후보의 식별자는 별도로 생성한다.

### 5.5 Place Verifier

좌표가 없거나 장소명만 추출된 경우 다음 순서로 보완한다.

1. URL에 포함된 장소명·주소·지역명 추출
2. TMAP POI 검색
3. 주소만 있으면 TMAP 지오코딩
4. 후보가 여러 개면 자동 선택하지 않고 최대 3개 표시
5. 사용자가 지도 마커와 주소를 확인한 뒤 등록

사용자가 확인하기 전에는 TMAP 대중교통 후보 평가를 시작하지 않는다.

### 5.6 Outbound Link Builder

정규화된 장소 좌표로 카카오맵·네이버 지도의 다음 동작을 생성한다.

- 지도에서 장소 보기
- 출발지에서 목적지까지 대중교통 길찾기
- 목적지 주변 검색

카카오맵은 공식 Web URL과 앱·모바일웹 Scheme을 각각 테스트한다. 네이버 지도는 공식 URL Scheme에 필수인 `appname`을 포함하고, 앱 미설치 시 웹 또는 설치 안내 fallback을 검증한다.

사용자가 카카오맵 링크로 가져온 장소라도 네이버 지도에서 열 수 있어야 한다. 이를 위해 외부 장소 ID가 아니라 내부의 WGS84 좌표와 장소명을 outbound 링크의 기본 입력으로 사용한다.

## 6. 가져오기 처리 흐름

1. 사용자가 URL을 붙여넣는다.
2. 클라이언트는 형식만 검사하고 서버에 원본 URL을 전송한다.
3. 서버가 공급자 allowlist와 URL Scheme을 확인한다.
4. 필요한 경우 안전하게 단축 URL을 확장한다.
5. 공급자 파서가 장소 ID·이름·좌표를 추출한다.
6. 좌표가 없으면 POI 검색 또는 지오코딩을 한 번 수행한다.
7. 서비스 지도에 마커·장소명·주소·출처를 표시한다.
8. 사용자가 확인하면 장소 후보로 등록한다.
9. 등록 후 기존 TMAP 요약정보 평가 큐에 `참여자 수 × 신규 후보 1개` 작업을 추가한다.

실패 시 사용자가 다시 처음부터 입력하게 하지 않는다.

| 실패 | 사용자 처리 |
|---|---|
| 지원하지 않는 지도 URL | 카카오맵·네이버 지도 링크만 지원한다고 안내 |
| 장소를 추출하지 못함 | 장소명 또는 주소 한 번 입력 |
| 같은 이름의 장소가 여러 개 | 지도와 주소를 포함한 최대 3개 후보 선택 |
| 외부 링크가 만료·삭제됨 | 장소명 검색 또는 지도에서 직접 위치 선택 |
| 악성·비허용 URL | 외부 요청 없이 차단 |

## 7. 외부로 보내기 처리 흐름

### 장소 탐색 단계

우리 서비스가 추천한 지역 카드에 다음 버튼을 제공한다.

- `카카오맵에서 주변 보기`
- `네이버 지도에서 주변 보기`

검색어는 사용자가 선택한 모임 목적과 지역명으로 만든다. 외부 지도에서 장소를 고른 뒤 공유 링크를 우리 서비스로 다시 가져온다.

### 후보 비교 단계

우리 서비스 안에 다음 정보가 남아야 한다.

- 참여자별 예상 이동시간
- 환승 횟수
- 예상 요금
- 그룹 평균·최대 이동시간
- 출발 위치와 후보를 잇는 간단한 경로선 또는 주요 교통수단

이 단계에서 사용자를 외부 지도로 강제 이동시키지 않는다.

### 확정·이동 단계

확정 카드에 다음 버튼을 제공한다.

- `카카오맵으로 길찾기`
- `네이버 지도에서 길찾기`

우리 서비스는 개인별 권장 출발시각과 예상 소요시간을 유지한다. 실시간 상세 경로와 실제 이동 안내는 외부 지도가 담당한다.

## 8. 경로 표시 수준 A/B PoC

외부 지도로 너무 일찍 보내면 서비스 가치가 약해지고, 모든 상세 경로를 자체 구현하면 비용과 개발량이 커진다. 두 안을 동일한 시나리오로 비교한다.

### A안: 요약만 내부 표시

- 시간·환승·요금·주요 수단만 표시
- 모든 상세 경로는 외부 지도에서 확인
- 장점: 구현과 TMAP 전체정보 호출 최소화
- 위험: 추천 근거가 약하게 느껴질 수 있음

### B안: 선택한 후보만 내부 경로선 표시

- 후보 목록에서는 요약만 표시
- 사용자가 후보 카드를 펼치면 TMAP 전체정보 1건을 지연 호출
- 지도에 실제 Polyline과 주요 교통수단 표시
- 상세 정류장 안내는 외부 지도에 위임
- 장점: 추천 근거와 시각적 wow 포인트 유지
- 위험: 펼칠 때 비용과 대기시간 추가

PoC 기본 추천은 B안이다. 전체 후보의 전체 경로를 미리 호출하지 않고 사용자가 선택한 후보에만 호출한다.

## 9. 호출량과 비용 관찰

장소 URL을 파싱하는 것 자체는 TMAP Transit 호출을 사용하지 않는다. 추가 호출 가능성은 다음과 같다.

| 상황 | 추가 호출 |
|---|---:|
| URL에 정확한 좌표가 있음 | 0건 |
| 장소명·주소만 있어 검증 필요 | POI 또는 지오코딩 1건 |
| 신규 장소 후보 등록 | 참여자 수만큼 TMAP 요약정보 |
| 후보 경로선 펼치기 | 해당 사용자·후보의 TMAP 전체정보 1건 |
| 외부 지도 버튼 클릭 | 우리 TMAP API 호출 없음 |

PoC 로그에는 `resolutionMethod`, fallback API 종류, 호출 수, 처리시간, 최종 성공 여부를 기록한다. TMAP 데이터 캐시는 기존 약관과 동일하게 24시간 미만으로 제한한다.

| 브라우저 | Chrome Android, Safari iOS, Chrome·Safari Desktop |
| 앱 상태 | 지도 앱 설치, 미설치 |
| 로그인 | 외부 지도 로그인, 비로그인 |

## 12. 단계별 실행 순서

### 1단계: 링크 표본 수집

- Android·iOS·PC에서 실제 공유 URL 수집
- 정답 장소명·주소·좌표 기록
- 개인정보와 현재 위치 제거

### 2단계: 순수 URL 파서

- 네트워크 호출 없이 해석 가능한 형식 구현·측정
- URL encoding, 쿼리 순서, 불필요 파라미터 변형 테스트

### 3단계: 안전한 단축 URL 확장

- allowlist와 redirect 제한 적용
- 정상·timeout·악성 redirect 테스트

### 4단계: 장소 보완과 확인 UI

- 좌표 누락 시 POI·지오코딩 fallback
- 중복 상호 선택과 지도 마커 확인

### 5단계: 외부 지도 열기

- 장소 보기, 주변 검색, 대중교통 길찾기 버튼
- 앱 설치·미설치와 모바일·PC 동작 기록

### 6단계: 경로 표시 A/B

- 요약만 표시하는 A안
- 선택 후보에 Polyline을 표시하는 B안
- 사용자가 후보를 이해하고 외부 길찾기로 자연스럽게 이동하는지 관찰

## 13. MVP 의사결정 규칙

- 카카오맵 실제 공유 링크 자동 성공률이 90% 이상이면 URL 직접 가져오기를 MVP에 포함한다.
- 네이버 지도 자동 성공률이 80% 미만이면 MVP에서 완전 자동을 약속하지 않고 `링크 붙여넣기 → 장소명/주소 확인` 방식으로 제공한다.
- 잘못된 장소가 자동 확정된 사례가 한 건이라도 있으면 모든 fallback 결과에 확인 단계를 유지한다.
- 앱 미설치 환경에서 안정적인 fallback을 만들 수 없는 공급자는 `앱에서 열기`와 `웹에서 보기` 버튼을 분리한다.
- A안만으로 추천 근거가 충분하지 않으면 B안을 채택하되 전체정보는 사용자 요청 시에만 호출한다.
- 외부 URL 형식 변경을 지속적으로 추적해야만 동작한다면 파서를 핵심 경로로 두지 않고 장소명·주소 입력을 항상 fallback으로 유지한다.

## 14. PoC 산출물

1. 개인정보가 제거된 URL 테스트 표본 JSON
2. 공급자별 URL parser·normalizer 프로토타입
3. 안전한 redirect resolver 테스트
4. 장소 확인용 최소 웹 화면
5. Android·iOS·PC outbound 링크 테스트 결과
6. 경로 요약 A안과 선택 후보 Polyline B안 비교 결과
7. 공급자·URL 유형별 PASS/FAIL 및 MVP 지원 범위 보고서

API 키와 실제 사용자 공유 링크 원본은 저장소에 커밋하지 않는다. 재현 가능한 비식별 예제만 산출물에 포함한다.
