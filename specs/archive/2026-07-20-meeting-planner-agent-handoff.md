# 모임 조율 서비스: 개발 에이전트 핸드오프 컨텍스트

## 0. 이 문서의 목적

이 문서는 과거 대화를 보지 못한 개발 에이전트가 제품 의도와 확정된 기술 결정을 오해하지 않고 spec-driven development를 시작하기 위한 컨텍스트다.

기능의 단일 상세 기준은 함께 제공되는 `2026-07-20-meeting-planner-feature-spec.md`다. 이 문서는 왜 그런 결정을 했는지, 무엇을 변경하면 안 되는지, 개발 전에 어떤 세부 spec을 작성해야 하는지를 설명한다.

## 1. 에이전트에게 전달할 한 문장

> 여러 참여자의 가능한 시간과 실제 출발 장소를 최소 입력으로 수집하고, 대중교통 도달권과 실제 이동시간으로 만나기 편한 후보를 제안한 뒤 각자의 출발 시각·경로까지 한 번에 정리하는 B2C 웹 서비스를 설계·개발한다.

## 2. 프로젝트 배경

- 대상 개발팀은 고등학생 멘티다.
- 수익화보다 시연 시의 WOW와 E2E 기술 경험이 중요하다.
- 화면, 백엔드, 데이터베이스, 외부 API, 공간 연산, 비동기 작업, 실시간 상태, 모니터링, 클라우드까지 전체 개발 사이클을 보여주고자 한다.
- 단순한 AI 기능이나 음식점 추천 서비스가 목적이 아니다.
- 핵심 기술 가치는 그룹 시간 집계, 공간 폴리곤 연산, 대중교통 API 조합, 공정성 점수, 설명 가능한 결과다.

## 3. 문제 정의

기존 모임 조율은 메신저에서 다음 질문을 반복한다.

- 누가 언제 가능한가?
- 각자 어디에서 출발하는가?
- 어디가 실제 대중교통 기준으로 공평한가?
- 최종적으로 무엇이 확정됐는가?

시간 투표 서비스와 중간 장소 서비스는 각각 존재한다. 이 제품의 차별점은 다음 세 가지를 하나의 흐름으로 결합하는 것이다.

1. 시간과 장소의 E2E 합의
2. 공통 대중교통 도달 가능 영역의 시각화
3. 시간·환승·요금과 확정 후 개인별 준비 정보를 한 번에 제공하는 생활 편의

`대중교통 데이터를 쓴다`는 사실만으로는 차별점이 아니다. 기존 국내 서비스도 이를 주장한다.

## 4. 최우선 UX 원칙

참여자의 필수 행동은 두 화면으로 제한한다.

1. 닉네임 + 실제 출발 장소
2. 가능한 날짜·시간

그 이후는 시스템과 방장이 처리한다.

- 초대 코드 직접 입력보다 코드가 포함된 링크가 기본이다.
- 최근 닉네임과 출발 장소를 브라우저에 기억한다.
- 시간은 오전·오후·저녁 빠른 선택을 기본으로 하고 30분 상세 그리드는 펼침 기능이다.
- 이동수단은 MVP에서 대중교통 고정이므로 입력받지 않는다.
- 출발 시각은 입력받지 않고 시간 확정 후 계산한다.
- 추천 장소는 기본 3개만 보여준다: 가장 공평, 가장 빠름, 환승이 편함.
- 투표는 필수가 아니다. 방장은 추천을 바로 확정하거나 의견이 갈릴 때만 별도 `결정` 탭에서 투표를 연다.
- 상세 화면은 `모임`, `후보`, `결정`의 세 탭을 사용한다.

## 5. 확정된 외부 API 역할

### TMAP 일반 API

- 지도 렌더링
- 장소명 POI 검색
- 주소를 입력했을 때 주소→WGS84 좌표 변환

장소 검색 결과가 좌표를 포함하면 지오코딩을 추가 호출하지 않는다.

### ODsay 대중교통 접근성 영역

- 참여자별 30·45·60분 버스+지하철 도달권 GeoJSON 생성
- 후보별 경로 조회에는 사용하지 않음
- 무료 Basic 한도 일 1,000건을 보호해야 함

### TMAP 대중교통 요약정보

- 후보별 실제 이동시간, 환승, 요금 평가
- `count=1`
- 사용하는 값: `totalTime`, `totalWalkTime`, `totalDistance`, `totalWalkDistance`, `transferCount`, `fare.regular.totalFare`, `pathType`

### TMAP 대중교통 전체정보

- 확정 후 사용자가 상세 경로를 열 때 지연 호출
- 장거리 fallback에서 대표 참여자 쌍의 환승역·터미널 추출
- `legs[].passShape.linestring`으로 버스·지하철·기차 Polyline 표시
- `WALK` 경로선은 표시하지 않음

## 6. 추천 엔진의 확정 동작

### 지역·인접 생활권 모드

1. 시간 확정과 모든 출발지 입력을 확인한다.
2. 가장 먼 출발지 간 직선거리로 첫 도달권 시간만 선택한다. 직선거리는 추천 점수에 사용하지 않는다.
3. 각 참여자에 대해 ODsay 도달권을 호출한다. PoC에서 수도권·경기 외곽·충주 30·60분 GeoJSON을 확인했다.
4. Turf.js 또는 PostGIS로 폴리곤 교집합을 구한다.
5. 교집합이 없으면 한 단계만 확대한다. PoC에서 강남·정왕 30분은 없고 60분은 존재했다. 모임당 최대 2라운드다.
6. MultiPolygon을 조각별 면적으로 정렬하고 작은 조각 제거 후 상위 3개를 먼저 탐색한다. PoC의 60분 교집합은 17개 조각이었다.
7. 상위 조각에서 TMAP POI 후보를 수집하고 결과가 없으면 다음 조각으로 확장한다.
8. 중복 제거 후 평가 후보를 최대 6개로 제한한다.
9. 참여자 수 × 후보 수만큼 TMAP 요약정보를 조회한다.
10. 기본 추천 3개를 만든다.

### 기본 후보 3개

```text
그룹 총 이동시간 = Σ totalTime
그룹 총 환승 횟수 = Σ transferCount
그룹 예상 교통비 = Σ totalFare
```

- 이동시간이 가장 적어요: 그룹 총 이동시간 최소
- 환승이 가장 적어요: 그룹 총 환승 횟수 최소
- 교통비가 가장 적어요: 그룹 예상 교통비 최소
- 같은 장소가 여러 기준에서 선정되면 차순위를 사용해 서로 다른 3개 후보를 우선 제공한다.
- 최장 이동시간과 편차는 분석용으로 보존하지만 사용자 메시지의 중심으로 삼지 않는다.
- 전역 최적을 보장한다고 표현하지 않는다. 외부 API와 제한 후보군을 이용한 현실적인 근사해다.

### 장거리 fallback

60분 도달권에도 교집합이 없을 때만 사용한다.

1. 기차역·고속버스터미널·시외버스터미널 후보 수집
2. 방향이 다른 대표 참여자 쌍 2~3개 선정
3. TMAP 전체 경로에서 환승역·터미널 추출
4. 후보 최대 6개로 제한
5. TMAP 요약정보로 전원 이동시간 평가
6. 사용자에게 장거리 근사 계산임을 명시

## 7. API 호출 보호 규칙

- 추천 계산은 자동 입력 이벤트가 아니라 방장 버튼으로 시작한다.
- 동일 입력 버전의 중복 계산을 금지한다.
- 계산 중 동일 요청을 잠근다.
- ODsay는 모임당 최대 2라운드, 참여자 최대 10명이다.
- TMAP 평가 후보는 최대 6개, 결과는 1개다.
- free 개발 요금제는 병렬 호출 시 429가 확인됐으므로 큐 동시성 1과 기본 3초 간격으로 시작한다.
- 429는 `Retry-After` 또는 지수 백오프+jitter로 최대 3회 재시도한다.
- 후불 종량제도 초당 호출 제한 해제를 뜻하지 않는다. 공급자에게 RPS/TPS·burst·동시 요청·API 간 제한 공유 여부를 확인한다.
- 큐 동시성, 초당 호출 수, 호출 간격, 모임별 예산은 환경설정으로 두고 유료 전환 직후 단계적 부하 PoC로 보정한다.
- 유료 환경에서도 공급자 한도보다 낮은 자체 rate limit, 요청 중복 제거, 일·월 비용 상한과 긴급 차단 스위치를 유지한다.
- 원본 ODsay 폴리곤은 서버 연산에 쓰고 지도용은 단순화해 전송한다.
- 한 참여자의 출발지만 변경되면 그 참여자×후보 경로만 다시 계산한다.
- 전체정보는 확정 전 일반 평가에 사용하지 않는다.
- 캐시 보존기간은 각 공급자 약관을 준수하고 24시간 미만으로 설정한다.
- 일일 사용량과 비용 추정을 `api_usage_logs`에 기록한다.
- 쿼터 부족 시 수동 후보 등록과 투표로 완료할 수 있어야 한다.

## 8. 명시적 비범위

다음 기능을 MVP에 다시 넣지 않는다.

- GPS 또는 실시간 위치 공유
- 별도 보행자 경로 API
- 도보 전용 및 자동차 이동수단
- 음식점·상품의 개인화 추천
- 결제
- AI 중심 추천
- 캘린더 연동
- 회원가입 강제

범위 변경이 필요하면 구현 전에 사용자 승인을 받는다.

## 9. 핵심 위험

### 공급자 간 결과 불일치

ODsay는 후보 영역 생성, TMAP은 최종 시간 평가에 사용한다. 두 공급자의 경로·배차·도보 처리 차이로 결과가 다를 수 있다. 도달권은 탐색용, 최종 표시는 TMAP 기준임을 UI에 명시한다.

ODsay isochrone에는 실제 출발 날짜·시간 입력이 없으므로 `그 시각에 반드시 60분 내 도착`으로 표현하지 않는다. 확정 시각의 경로 가능성과 시간은 TMAP에서 최종 검증한다.

### free 요금제 계산 지연

5명×6후보는 30건이다. 3초 간격 직렬 처리 시 약 90초가 걸릴 수 있으므로 개발 환경에서는 비동기 작업, 진행률, 부분 결과가 필수다. 운영 유료 요금제의 허용 RPS를 확인하기 전 즉시 계산 UX를 약속하지 않는다.

### 입력 부담

정확성을 위해 입력을 늘리지 않는다. 참여자는 닉네임·출발지·가능 시간만 입력한다. 투표는 선택적이다.

### 지방 데이터 품질

전국 지원을 주장하기 전에 실제 지역 표본을 테스트한다. 최소 테스트 지역은 수도권, 광역시, 지하철 없는 중소도시, 인접 시·도, 60분 초과 장거리다.

### 전역 최적 오해

후보군 내 최적일 뿐 전국 모든 장소의 전역 최적이 아니다. 추천 이유와 한계를 함께 표시한다.

### 개인정보

정확한 집 주소가 아니라 사용자가 선택한 공개 장소명을 다른 참여자에게 표시한다. 외부 API 키는 서버에만 둔다.

## 10. 개발 우선순위

### P0: 수동으로 완결되는 사용자 흐름

- 최소 입력 모임 생성
- 초대 링크와 비회원 참여
- 닉네임·출발지 통합 입력
- 빠른 시간 입력과 히트맵
- 추천 시간 바로 확정 또는 선택적 투표
- 수동 장소 후보, 바로 확정 또는 선택적 투표
- 확정 공유 페이지

### P1: 목표 MVP 추천 엔진

- ODsay 도달권
- GeoJSON 교집합
- 공통 영역 시각화
- TMAP POI 후보
- TMAP 요약정보
- 세 가지 추천 기준과 설명
- 호출량·비용 로그

### P2: 고도화

- TMAP 경로 Polyline 지연 조회
- 장거리 fallback
- 권장 출발 시각
- WebSocket
- 운영 대시보드

P0를 완성하고 검증하기 전 P1을 병렬로 구현하지 않는다. 외부 API를 붙이기 전에 수동 후보를 이용한 E2E 흐름이 동작해야 한다.

## 11. Spec-driven development 절차

개발 에이전트는 코드를 쓰기 전에 다음 순서로 문서를 작성하고 검토받아야 한다.

1. 저장소와 기존 기술 스택, 코딩 규칙, 실행 방법 조사
2. `product-requirements.md`: 사용자 스토리, 범위, 성공 기준
3. `domain-model.md`: 엔터티, 상태 전이, 권한, 불변조건
4. `api-contracts.md`: 내부 REST/WebSocket과 외부 API 어댑터 계약
5. `recommendation-engine.md`: 도달권, 교집합, 후보 생성, 점수, fallback
6. `ux-flow.md`: 방장·참여자 화면 흐름과 빈·로딩·오류 상태
7. `observability.md`: 쿼터·비용·실패율·작업 큐 모니터링
8. `test-plan.md`: 단위·통합·계약·E2E 테스트 사례
9. 구현 계획을 수직 슬라이스 단위로 작성
10. 사용자 승인 후 테스트 우선으로 구현

각 spec에는 반드시 다음을 포함한다.

- 목표와 비목표
- 입력·출력
- 정상 흐름
- 오류·빈 상태
- 권한
- 데이터 보존기간
- 수락 기준
- 미해결 질문

## 12. 권장 수직 슬라이스

1. 모임 생성 → 링크 참가 → 참여자 표시
2. 시간 입력 → 집계 → 방장 확정
3. 출발 장소 검색 → 선택 → 지도 마커
4. 수동 장소 후보 → 방장 확정·선택적 투표
5. ODsay 어댑터 → 도달권 저장 → 교집합
6. TMAP POI → 후보 6개
7. TMAP 요약 → 세 가지 추천 후보
8. 오류 fallback → 쿼터·비용 로그
9. 상세 경로 지연 조회

각 슬라이스는 UI, API, DB, 테스트, 로그까지 포함한 E2E 완료 단위여야 한다.

## 13. 에이전트가 먼저 확인할 미해결 질문

다음은 구현 전에 저장소와 사용자에게 확인해야 한다.

1. 기존 코드베이스와 확정 기술 스택이 있는가?
2. 웹만 지원하는가, 반응형 모바일 웹 또는 앱까지 포함하는가?
3. 지도 SDK는 TMAP으로 확정했는가?
4. TMAP 검색 시각을 확정 날짜·시간으로 전달했을 때 낮·심야 경로 차이가 정상 반영되는가?
5. 불규칙한 폴리곤 내부 POI 수집을 centroid·bbox·grid 중 어떤 방식으로 구현할 것인가?
6. 투표 마감·방장 위임·참여자 강퇴가 MVP에 필요한가?
7. 공통 영역 내 허용할 POI 카테고리와 지역별 fallback 목록은 무엇인가?
8. 초기 배포 환경과 예산 한도는 얼마인가?
9. TMAP이 계약상 보장하는 요약·전체 API별 RPS/TPS, burst, 동시 요청 수와 두 API의 제한 공유 여부는 무엇인가?

ODsay 서버 키·IP 등록, isochrone 반환, TMAP Transit 사용 신청과 핵심 응답 필드는 PoC에서 확인됐으므로 미해결 질문이 아니다.

답이 없을 경우 임의로 범위를 확대하지 말고, P0를 구현할 수 있는 최소 가정을 명시해 승인받는다.

## 14. 에이전트에게 그대로 전달할 실행 프롬프트

```text
첨부된 기능 명세서와 핸드오프 컨텍스트를 제품 요구사항의 기준으로 사용하라.

목표는 여러 사용자의 가능한 시간과 수동 출발 장소를 최소 입력으로 수집하고, ODsay 도달권과 TMAP 이동시간으로 만나기 편한 장소 후보를 제안한 뒤 각자의 출발 준비 정보까지 정리하는 B2C 웹 서비스다.

즉시 코드를 작성하지 말라. 먼저 저장소를 조사하고 아래 spec을 순서대로 작성하라.
1) product requirements
2) domain model 및 상태 전이
3) 내부·외부 API 계약
4) 추천 엔진 상세 설계
5) UX 흐름 및 오류 상태
6) 테스트·관측성 계획
7) 수직 슬라이스 구현 계획

모호한 사항은 목록화하고 제품 방향을 바꾸는 질문만 사용자에게 하나씩 물어라. GPS, 도보 전용 API, 자동차, 음식점 개인화 추천, 결제, AI 중심 기능은 MVP 범위 밖이다. 투표는 필수가 아니며 방장이 추천을 바로 확정하거나 필요할 때만 연다. 참여자의 필수 화면은 닉네임·출발지와 가능한 시간의 두 화면이다.

P0 수동 E2E 흐름을 먼저 완성하고, 검증 후 P1 추천 엔진을 붙이는 계획을 제안하라. 각 요구사항에 고유 ID와 Given/When/Then 수락 기준을 부여하고, spec과 테스트와 구현 사이의 추적성 표를 유지하라.
```

## 15. 완료 정의

개발은 단순히 화면이 보이는 것으로 완료되지 않는다.

- 요구사항 ID가 테스트와 연결됨
- 정상·오류·쿼터 초과 흐름 검증
- 외부 API를 테스트 더블로 대체할 수 있음
- 후보 집계 기준과 동점 처리 규칙이 설정·테스트로 관리됨
- 동일 입력 버전의 중복 계산이 방지됨
- API 호출량과 비용이 관측됨
- 수동 fallback으로 모임을 끝낼 수 있음
- 배포 환경에서 핵심 E2E 시나리오가 통과함
