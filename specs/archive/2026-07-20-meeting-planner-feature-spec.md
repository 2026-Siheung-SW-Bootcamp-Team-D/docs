# 모임 조율 서비스 기능 명세서

- 문서 버전: 1.3
- 작성일: 2026-07-20
- 대상: 고등학생 개발 프로젝트
- 제품 형태: B2C 웹 서비스

## 1. 제품 개요

### 1.1 제품 목표

여러 참여자의 가능한 시간과 실제 출발 장소를 최소한의 입력으로 수집하고, 대중교통으로 만나기 편한 시간과 장소를 제안한다. 모임 확정 후에는 각 참여자가 다시 검색하지 않아도 출발 시각·이동시간·경로를 확인할 수 있게 한다. 공정성은 사용자 메시지가 아니라 내부 정렬 기준 중 하나로 사용한다.

### 1.2 핵심 사용자 가치

1. 메신저에서 반복되는 시간·장소 논의를 한 화면에서 처리한다.
2. 단순한 지리적 중앙이 아니라 실제 대중교통 도달 가능성과 이동시간을 사용한다.
3. 후보별 그룹 총 이동시간, 평균 이동시간, 환승 합계, 예상 교통비를 비교한다.
4. 시간 입력과 투표 결과가 실시간으로 변하는 시각적 경험을 제공한다.
5. 확정 결과를 개인별 약속 준비 카드와 지도 경로로 재가공한다.

### 1.3 핵심 사용자 흐름

1. 방장이 모임을 생성한다.
2. 초대 코드 또는 링크를 공유한다.
3. 참여자가 닉네임, 가능한 시간, 실제 출발 장소를 입력한다.
4. 시스템이 가능한 시간대를 집계한다.
5. 방장이 추천 시간을 바로 확정하거나 필요할 때만 투표를 시작한다.
6. 시스템이 참여자별 대중교통 도달권의 공통 영역을 계산한다.
7. 공통 영역 안에서 모임 장소 후보를 생성한다.
8. 실제 대중교통 이동시간으로 후보를 평가한다.
9. 방장이 추천 장소를 바로 확정하거나 필요할 때만 투표를 시작한다.
10. 확정 카드와 참여자별 대중교통 경로를 제공한다.

## 2. 범위

### 2.1 목표 MVP 포함(P0+P1)

- 모임 생성 및 초대
- 비회원 닉네임 참여
- 진행 중인 모임 목록
- 참여자 목록 및 입력 상태
- 가능한 시간 빠른 입력, 집계, 선택적 투표, 확정
- 장소명 또는 주소로 출발 장소 수동 입력
- 지도에 참여자 출발 장소 표시
- ODsay 대중교통 도달권 교집합 계산
- TMAP POI를 이용한 장소 후보 생성
- TMAP 대중교통 요약정보를 이용한 후보 평가
- 기본 장소 후보 3개, 선택적 투표 및 확정
- 확정 모임 공유 페이지

### 2.2 MVP 제외

- GPS 현재 위치 입력
- 당일 실시간 위치 공유
- 별도 보행자 경로 API
- 도보 전용 이동수단
- 자동차 경로
- 음식점·상품의 개인화 추천
- 결제 및 수익화
- AI만으로 생성하는 추천
- 외부 캘린더 연동

### 2.3 후속 확장

- 확정 장소까지의 버스·지하철·기차 경로선 표시
- 개인별 권장 출발 시각
- 회원가입 및 여러 기기 동기화
- 푸시 알림
- 운영·API 사용량 대시보드
- GPS와 당일 ETA

## 3. 사용자 역할

| 역할 | 권한 |
|---|---|
| 방장 | 모임 생성·수정, 투표 시작·종료, 추천 계산, 최종 시간·장소 확정 |
| 참여자 | 초대 링크 참가, 개인 정보 입력·수정, 방장이 연 시간·장소 투표 참여, 확정 정보 확인 |

MVP에서는 회원가입을 강제하지 않는다. 참여자 접근 토큰을 발급하고 브라우저 로컬 저장소에 모임 ID와 함께 저장한다.

| `COLLECTING` | 참여자와 출발지·가능 시간 수집 중 |
| `TIME_VOTING` | 방장이 선택적으로 연 시간 후보 투표 중 |
| `TIME_CONFIRMED` | 모임 시간 확정, 장소 계산 가능 |
| `PLACE_CALCULATING` | 도달권과 후보 장소 계산 중 |
| `PLACE_VOTING` | 방장이 선택적으로 연 장소 후보 투표 중 |
| `CONFIRMED` | 시간과 장소 확정 |
| `COMPLETED` | 모임 종료 |
| `CANCELLED` | 모임 취소 |

## 5. 기능 명세

### F01. 새 모임 만들기

방장이 최소 정보로 모임방을 생성한다. 생성 이후 선택 정보를 추가할 수 있다.

| 항목 | 필수 | 규칙 |
|---|---:|---|
| 모임 이름 | 필수 | 2~30자 |
| 후보 날짜 범위 | 필수 | 시작일은 종료일 이전 |
| 모임 목적 | 선택 | 생성 후 입력 가능, 100자 이하 |
| 1인 예산 | 선택 | 생성 후 입력 가능, 추천 점수에는 사용하지 않음 |
| 예상 소요 시간 | 선택 | 기본 2시간, 30분 단위 |
| 최대 참여자 | 선택 | 기본 10명, 최대 10명 |

생성 결과는 모임 ID, 6자리 초대 코드, 초대 링크, 방장 접근 토큰이다.

### F02. 초대 링크로 참여하기

기본 진입 수단은 코드가 포함된 초대 링크다. 홈 화면의 코드 직접 입력은 보조 수단으로 제공한다.

- 닉네임은 모임 안에서 중복될 수 없다.
- 만료·취소·최대 인원 초과 모임에는 참가할 수 없다.
- 브라우저에 최근 닉네임이 있으면 `이 닉네임으로 참여`를 제안한다.
- 참여 성공 후 닉네임·출발 장소 통합 화면과 가능한 시간 화면을 순서대로 보여준다.

### F03. 진행 중인 모임 확인하기

사용자가 참가한 모임을 상태별 카드로 표시한다. 모임 상세는 `모임`, `후보`, `결정`의 세 탭으로 구성한다.

- 모임 이름
- 상태
- 참여 인원
- 투표 마감 시각
- 사용자가 완료하지 않은 작업
- 확정된 시간과 장소

비회원 MVP에서는 현재 브라우저에 저장된 모임만 표시한다.

- `모임`: 현재 상태, 참여자 진행률, 내 입력, 확정 결과
- `후보`: 추천 시간과 기본 장소 후보 3개
- `결정`: 방장이 시작한 시간 또는 장소 투표만 표시

### F04. 참여자 목록

닉네임, 방장 여부, 사용자 구분 색상과 다음 입력 상태를 표시한다.

- 시간 입력 완료 여부
- 출발 장소 입력 완료 여부
- 시간 투표 여부
- 장소 투표 여부

다른 참여자의 정확한 주소 대신 사용자가 선택한 공개 장소명을 표시한다.

### F05. 가능한 시간 입력

참여자는 방장이 지정한 날짜 범위 안에서 가능한 시간대를 선택한다.

- 빠른 입력: 날짜 선택 후 오전·오후·저녁 구간 버튼
- 상세 입력: 필요한 사용자만 30분 단위 그리드 펼치기
- 기본 단위: 30분
- 드래그로 연속 구간 선택
- 다시 선택하여 해제
- 입력 변경 시 집계 결과 갱신
- 모임 예상 소요 시간을 만족하지 못하는 짧은 구간 제외

서버는 각 시간 슬롯의 가능 인원과 참여자 ID 목록을 집계한다.

### F06. 시간 후보 및 투표

후보 우선순위는 다음과 같다.

1. 전원이 가능한 연속 시간대
2. 가능한 인원이 가장 많은 시간대
3. 동일 인원이면 더 이른 시간대

전원이 가능한 후보가 명확하면 방장이 바로 확정할 수 있다. 의견 수렴이 필요할 때만 최대 5개 후보로 시간 투표를 시작한다. 참여자는 복수 선택할 수 있고 방장이 최종 시간을 확정한다.

### F07. 출발 장소 수동 입력

GPS 대신 실제 출발 예정 장소를 수동으로 입력한다. 닉네임과 출발 장소는 한 화면에서 받는다.

입력 방법:

1. 학교·역·건물·랜드마크 등 장소명 검색
2. 검색 결과에서 정확한 장소 선택
3. 결과가 없으면 도로명 주소 입력

브라우저에 저장된 최근 출발 장소가 있으면 검색창 위에 빠른 선택 버튼으로 제공한다. MVP 이동수단은 대중교통으로 고정하고 별도 선택 입력은 노출하지 않는다.

저장 항목:

- 외부 장소 ID
- 장소명
- 공개 표시명
- 주소
- WGS84 경도·위도
- 선택 이동수단: MVP에서는 대중교통 고정

기본 장소 검색은 TMAP POI API를 사용한다. 장소명 검색 결과에 좌표가 포함되므로 별도 지오코딩이 필요 없다. 주소를 직접 입력한 경우에만 TMAP 지오코딩을 사용한다. GPS를 사용하지 않으므로 좌표→주소 변환은 MVP에서 제외한다.

PoC에서 TMAP POI 응답의 장소 좌표와 카테고리, TMAP 지오코딩의 WGS84 좌표 반환을 확인했다. 외부 DTO의 `frontLon`, `noorLon` 등 공급자 필드명은 어댑터에서 내부 `longitude`, `latitude`로 정규화한다. 존재하지 않는 주소는 HTTP 400과 `A2C500` 오류가 확인됐으며 사용자에게 주소 수정·재검색 화면을 제공한다.

### F08. 지도

지도에는 다음을 표시한다.

- 참여자 출발 장소: 사용자별 색상 마커
- ODsay 도달권: 사용자별 반투명 폴리곤
- 공통 도달 가능 영역: 강조 폴리곤
- 장소 후보: 카테고리별 마커
- 확정 장소: 별도 강조 마커

정확한 집 주소 노출을 방지하기 위해 사용자가 검색 결과에서 선택한 공개 장소만 마커로 표시한다.

### F09. 추천 장소 계산 요청

다음 조건을 만족할 때 방장이 계산을 시작한다.

- 모임 시간이 확정됨
- 참여자 2명 이상
- 모든 참여자가 출발 장소 입력 완료
- 다른 계산이 진행 중이지 않음

사용자가 입력할 때마다 자동으로 외부 API를 호출하지 않는다. 동일한 입력 버전에 대해 중복 계산 요청을 차단한다.

### F10. 대중교통 도달권 계산

ODsay `searchPubTransIsochrone` API를 참여자별로 호출한다.

요청값:

- `x`, `y`: 출발 장소 WGS84 좌표
- `searchTime`: 30, 45 또는 60분
- `searchMethod=4`: 버스+지하철

응답에서 WGS84 GeoJSON 도달권 폴리곤을 사용한다. 참여자별 폴리곤의 교집합은 자체 서버에서 Turf.js 또는 PostGIS로 계산한다. PoC에서 수도권·경기 외곽·충주 도달권과 30·60분 응답을 확인했고, 60분 응답은 약 56KB였다.

ODsay 도달권 요청에는 실제 출발 날짜·시간 파라미터가 없으므로 결과를 `그 시각에 반드시 도착 가능한 영역`으로 표현하지 않는다. 도달권은 후보 탐색 영역이며, 확정된 모임 날짜·시간의 최종 이동 가능 여부와 소요시간은 TMAP 경로 응답으로 검증한다.

초기 검색시간은 가장 먼 참여자 사이의 직선거리로 API 호출 단계만 추정한다.

| 최대 직선거리 | 첫 검색시간 |
|---:|---:|
| 15km 이하 | 30분 |
| 15~35km | 45분 |
| 35km 초과 | 60분 |

직선거리는 추천 순위에 사용하지 않는다. 첫 교집합이 없으면 한 단계만 자동 확대한다. 모임당 자동 도달권 계산은 최대 2라운드로 제한한다.

### F11. 공통 영역과 장소 후보 생성

교집합이 여러 폴리곤으로 분리되면 모든 조각을 POI 탐색 대상으로 사용하지 않는다. PoC에서 강남 60분과 정왕 60분 교집합이 17개 조각으로 나왔으므로 다음 규칙을 적용한다.

1. MultiPolygon을 개별 Polygon으로 분해한다.
2. 각 조각의 면적과 전체 면적 대비 비율을 계산한다.
3. 매우 작은 조각을 설정 임곗값으로 제거한다.
4. 면적 상위 3개 조각을 기본 탐색 영역으로 선택한다.
5. 상위 영역에서 유효한 POI를 찾지 못하면 다음 순위 조각을 순차 탐색한다.

원본 폴리곤은 서버 교집합 계산에 사용하고, 지도 전송용 데이터는 topology를 보존하는 단순화를 적용한다. 원본·단순화 결과의 면적 차이와 좌표 수를 기록한다.

TMAP POI API를 이용해 공통 영역 안에서 다음 후보를 검색한다.

1. 기차역
2. 지하철역
3. 고속·시외버스터미널
4. 주요 버스 환승시설
5. 복합문화·상업시설
6. 사용자가 직접 추가한 장소

POI 결과의 좌표가 공통 영역 안에 포함되는지 자체 서버에서 검사한다. 동일 장소와 같은 역의 중복 결과를 제거하고 최종 평가 후보를 최대 6개로 제한한다.

### F12. 장거리·교집합 없음 fallback

60분 도달권에서도 공통 영역이 없으면 `LONG_DISTANCE` 계산 모드로 전환한다.

1. TMAP에서 기차역·고속버스터미널·시외버스터미널 후보를 수집한다.
2. 지리적으로 멀고 방향이 다른 대표 참여자 쌍 2~3개를 고른다.
3. TMAP 대중교통 전체정보에서 주요 환승역·터미널을 추출한다.
4. 중복 제거 후 후보를 최대 6개로 제한한다.
5. 모든 참여자에서 각 후보까지 TMAP 요약정보를 조회한다.

화면에는 `60분 내 공통 도달 지역이 없어 장거리 교통 거점을 기준으로 계산했습니다`라고 표시한다. 이 방식은 전역 최적을 보장하지 않는 근사 방식임을 명시한다.

### F13. 후보별 실제 이동시간 평가

TMAP 대중교통 요약정보 API를 `count=1`로 호출한다.

확정된 모임 날짜·시간이 경로 API의 검색 시각으로 지원되는지 별도 계약 테스트를 수행한다. 지원될 경우 모든 후보는 같은 기준 시각으로 조회한다.

사용 응답값:

| 필드 | 용도 |
|---|---|
| `totalTime` | 전체 예상 이동시간 |
| `totalWalkTime` | 대중교통 경로에 포함된 총 도보시간, 별도 도보 API 호출 없음 |
| `totalDistance` | 전체 이동거리 |
| `totalWalkDistance` | 총 도보거리 |
| `transferCount` | 환승 페널티 |
| `fare.regular.totalFare` | 참고 요금 표시 |
| `pathType` | 경로 유형 표시 |

경로가 없는 참여자가 있으면 기본적으로 해당 후보를 제외한다. 모든 후보에 경로 누락이 있으면 도달 가능한 인원이 많은 순으로 표시하고 경고한다.

### F14. 후보 집계와 기본 추천 3개

후보별 참여자 경로를 생활 편의 지표로 집계한다.

```text
그룹 총 이동시간 = Σ 참여자 totalTime
그룹 총 환승 횟수 = Σ 참여자 transferCount
그룹 예상 교통비 = Σ 참여자 totalFare
평균 이동시간 = 그룹 총 이동시간 / 경로가 있는 참여자 수
```

불투명한 종합 점수 하나로 순위를 정하지 않고 사용자가 이해하기 쉬운 다음 세 가지 기본 후보를 제공한다.

| 추천 유형 | 기준 |
|---|---|
| 이동시간이 가장 적어요 | 그룹 총 이동시간 최소 |
| 환승이 가장 적어요 | 그룹 총 환승 횟수 최소, 동률이면 총 이동시간 |
| 교통비가 가장 적어요 | 그룹 예상 교통비 최소, 동률이면 총 이동시간 |

각 후보에는 그룹 총 이동시간, 평균 이동시간, 환승 합계, 예상 교통비 합계와 규칙 기반 추천 이유를 표시한다. 최장 이동시간과 편차는 내부 분석과 상세 정보에 보존하지만 제품의 대표 메시지로 사용하지 않는다.

서로 다른 기준에서 같은 장소가 선정되면 중복을 제거하고 해당 기준의 차순위 장소를 넣는다. 기본 화면에는 3개만 노출하며 나머지는 `다른 후보 보기`에서 펼친다.

### F15. 장소 투표

- 장소 투표는 필수가 아니다. 방장은 추천 장소를 바로 확정하거나 투표를 시작한다.
- 기본 추천 3개를 투표 선택지로 자동 등록
- 참여자별 단일 선택
- 방장이 투표 시작·종료
- 실시간 투표 인원 표시
- 동점이면 방장이 최종 선택
- 사용자가 직접 추가한 장소도 같은 평가를 거친 뒤 후보로 등록

### F16. 최종 일정 확정

방장이 시간과 장소를 확정하면 모임 상태를 `CONFIRMED`로 변경한다.

- 확정 시간·장소 저장
- 참여자별 예상 대중교통 시간 표시
- 권장 출발 시각 계산: 모임 시각 - 예상 이동시간 - 10분
- 확정 공유 페이지 생성

### F17. 경로 시각화

P1 기능으로 TMAP 대중교통 전체정보 API를 사용한다.

사용 응답값:

- `legs[].mode`
- `legs[].sectionTime`
- `legs[].start`, `legs[].end`
- `legs[].passShape.linestring`

`passShape.linestring`을 좌표 배열로 변환해 지도 Polyline으로 표시한다. `WALK` 구간은 그리지 않고 버스·지하철·기차 구간만 이동수단별 색상으로 표시한다. 출발지에서 첫 정류장까지는 `도보 N분` 텍스트만 표시한다.

호출량을 줄이기 위해 다음 시점에만 전체정보를 조회한다.

1. 장거리 fallback의 대표 참여자 쌍 2~3개
2. 장소 확정 후 사용자가 자신의 상세 경로를 열 때

### F18. 모임 예정 카드 공유

공유 페이지에는 다음을 표시한다.

- 모임 이름과 목적
- 날짜·시간
- 장소명과 주소
- 지도 썸네일
- 참여 인원
- 개인별 예상 이동시간과 권장 출발 시각
- 모임 링크

MVP에서는 HTML 공유 페이지와 Open Graph 메타데이터를 제공한다. 이미지 다운로드는 P1이다.

## 6. 외부 API 명세

### 6.1 TMAP 일반 API

용도:

- 지도 SDK
- POI 장소 검색
- 주소→좌표 지오코딩

POI에서 사용할 값은 장소 ID, 명칭, 주소, 경도·위도, 카테고리다. 장소 검색 결과에 좌표가 있으면 지오코딩을 추가 호출하지 않는다.

### 6.2 ODsay 대중교통 접근성 영역 API

엔드포인트:

```text
GET/POST https://api.odsay.com/v1/api/searchPubTransIsochrone
```

용도는 참여자별 30·45·60분 대중교통 도달권 GeoJSON 생성이다. 후보별 길찾기에는 사용하지 않는다.

### 6.3 TMAP 대중교통 요약정보 API

엔드포인트:

```text
POST https://apis.openapi.sk.com/transit/routes/sub
```

용도는 후보별 실제 이동시간·환승·요금 평가다. 참여자 수 × 후보 수만큼 호출하되 후보는 최대 6개, 결과 수는 1개로 제한한다.

### 6.4 TMAP 대중교통 전체정보 API

엔드포인트:

```text
POST https://apis.openapi.sk.com/transit/routes
```

용도는 최종 경로선과 장거리 fallback의 주요 환승 거점 추출이다. 모든 후보 평가에 사용하지 않는다.

## 7. API 호출 제한 및 비용 통제

### 7.1 ODsay

- 모임당 자동 도달권 계산 최대 2라운드
- 참여자 최대 10명
- 방장만 재계산 가능
- 입력 버전이 동일하면 중복 호출 금지
- 계산 중 중복 버튼 요청 잠금
- 일일 쿼터 80% 도달 시 자동 재계산 제한

5인 모임은 1라운드 5건, 최대 2라운드 10건을 사용한다.

### 7.2 TMAP 요약정보

- 후보 최대 6개
- 결과 `count=1`
- 5인 모임 기준 최대 30건
- 한 명의 출발지만 바뀌면 해당 참여자의 후보 6건만 재계산
- 경로 결과 캐시는 API 약관에 맞춰 24시간 미만으로 제한
- free 개발 요금제에서는 동시 호출 시 429 `THROTTLED`가 확인됐으므로 외부 호출을 작업 큐에서 제어
- free 개발 환경 기본 동시성 1, 최소 호출 간격 3초로 시작하되 환경설정으로 관리
- 429 응답의 `Retry-After`가 있으면 우선 적용하고, 없으면 지수 백오프와 jitter로 최대 3회 재시도
- 후불 종량제는 호출 건별 과금 방식일 뿐 초당 호출 제한 해제를 보장하지 않는다. 공식 공통 오류 문서도 429 `THROTTLED`를 초당 호출 건수 초과로 정의한다.
- 유료 전환 전에 TMAP Data Biz팀에 요약정보·전체정보의 RPS/TPS, burst, 동시 요청 수, 두 API의 제한 공유 여부를 확인한다.
- 유료 전환 후 제한된 호출 예산으로 단계적 부하 PoC를 수행하고, 실제 허용량보다 낮은 서비스 자체 제한을 유지한다.
- 큐 동시성, 초당 호출 수, 최소 간격, 모임당 호출 예산은 배포 없이 변경할 수 있는 환경설정으로 관리한다.
- 재시도 폭주와 중복 과금을 막기 위해 요청 deduplication, 모임별 예산, 일·월 비용 상한, 긴급 차단 스위치를 둔다.
- 계산 진행률과 부분 완료 후보를 클라이언트에 전달

### 7.3 TMAP 전체정보

- 확정 전 일반 후보 평가에는 사용 금지
- 상세 경로를 연 사용자에게 지연 호출
- 장거리 fallback은 대표 쌍 최대 3건

### 7.4 예시 비용

사용자 제공 단가인 요약 0.55원, 전체 0.88원을 기준으로 한 5인·6후보 모임:

```text
요약정보: 5 × 6 × 0.55원 = 16.5원
확정 후 전체정보 5건: 5 × 0.88원 = 4.4원
합계: 20.9원
```

ODsay는 무료 한도 안에서 도달권 생성에만 사용한다.

## 10. 오류 및 예외 처리

| 상황 | 처리 |
|---|---|
| 잘못된 초대 코드 | 참가 불가 메시지 |
| 장소 검색 결과 없음 | 주소 입력 또는 다른 검색어 안내 |
| 존재하지 않는 주소, TMAP `A2C500` | 입력값을 유지하고 주소 수정·재검색 안내 |
| 일부 ODsay 호출 실패 | 재시도 1회 후 해당 참여자 표시, 전체 계산 중단 |
| 도달권 교집합 없음 | 한 단계 확대 후 장거리 fallback |
| 공통 영역 내 POI 없음 | 사용자 직접 후보 추가 또는 장거리 fallback |
| 일부 TMAP 경로 없음 | 후보 제외 및 이유 표시 |
| TMAP 429 `THROTTLED` | 큐에서 간격 조절 후 백오프 재시도, 최종 실패 시 계산 이어받기 안내 |
| 모든 후보 경로 없음 | 추천 실패, 수동 장소 투표 제공 |
| API 쿼터 부족 | 자동 계산 중지, 기존 후보 또는 수동 후보 제공 |
| 중복 계산 요청 | 기존 작업 상태 반환 |

항상 수동 장소 후보와 수동 투표로 완료할 수 있는 fallback을 제공한다.

## 11. 개인정보 및 보안

- GPS를 수집하지 않는다.
- 사용자가 선택한 출발 장소만 저장한다.
- 다른 참여자에게 상세 주소 대신 공개 장소명을 표시한다.
- API 키는 서버 환경변수와 비밀 저장소에 보관한다.
- 외부 API를 브라우저에서 직접 호출하지 않는다.
- 접근 토큰은 해시하여 저장한다.
- 경로·도달권 캐시는 공급자 약관에 맞춘 만료시간을 설정한다.

- 자동차 이동수단

## 13. MVP 성공 기준

- 5명의 사용자가 초대 링크로 같은 모임에 참가할 수 있다.
- 모든 참여자가 출발 장소와 가능한 시간을 입력할 수 있다.
- 전원 또는 최다 인원이 가능한 시간 후보가 표시된다.
- 도달권 계산이 모임당 최대 2라운드 안에서 종료된다.
- 공통 영역과 기본 추천 후보 3개가 표시되고 추가 후보를 펼칠 수 있다.
- 각 후보에 평균·최장 이동시간과 환승 정보가 표시된다.
- 이동시간·환승·교통비 기준의 기본 후보가 구분된다.
- 방장이 추천 시간·장소를 바로 확정하거나 별도 결정 탭에서 투표를 열 수 있다.
- API 실패 시에도 수동 후보 투표로 모임을 완료할 수 있다.

## 15. 명시적 한계

- 추천 결과는 전국 모든 장소를 완전 탐색한 수학적 전역 최적해가 아니다.
- ODsay 도달권과 TMAP 후보 경로를 결합한 현실적인 근사해다.
- API 공급자별 데이터 갱신 시점과 경로 알고리즘 차이로 결과가 다를 수 있다.
- 지방의 매우 낮은 배차 빈도와 60분을 넘는 장거리 이동은 장거리 fallback으로 처리한다.
- ODsay 도달권은 출발 날짜·시간에 따른 실제 운행 여부를 보장하는 데이터가 아니라 후보 영역 생성용이다.
- 추천은 최종 결정을 대신하지 않으며 참여자 투표와 방장 확정으로 종료한다.

## 16. PoC 검증 결과

2026-07-20 사전 검증에서 다음을 실데이터로 확인했다.

- ODsay 강남·정왕·충주 30·60분 도달권 GeoJSON 정상 반환
- GeoJSON WGS84 좌표계와 약 56KB 규모의 60분 응답
- 강남 60분∩정왕 60분 교집합 93.4km², 약 65ms 계산
- 강남 30분∩정왕 30분 교집합 없음, 한 단계 확대 필요
- 강남 60분∩충주 60분 교집합 없음, 장거리 fallback 조건 확인
- 60분 교집합이 17개 분리 조각으로 반환돼 상위 영역 필터 필요
- TMAP POI 좌표·카테고리와 지오코딩 WGS84 응답
- TMAP 요약정보의 시간·도보·환승·요금 필드
- TMAP 전체정보의 `legs[].mode`, `legs[].passShape.linestring`
- 수도권·경기 외곽·충주→청주 시외버스·서울→천안 장거리 경로
- free 요금제 병렬 호출 시 429와 3초 간격 재시도 후 성공

PoC가 검증하지 않은 다음 항목은 구현 단계에 맞춰 추가 검증한다.

### 구현 전·무료 환경

1. 불규칙한 교집합 폴리곤 내부에서 centroid·bbox grid별 TMAP POI 수집률, 중복률, 호출 수 비교
2. TMAP 검색 시각 파라미터를 적용한 평일 낮·심야·주말 결과 차이와 막차 이후 무경로 처리
3. GeoJSON 단순화 수준별 지도 품질, payload 크기, 교집합 면적 오차
4. TMAP 전체 경로 Polyline의 실제 브라우저 렌더링·레이스 애니메이션 성능
5. free 환경 429 응답의 헤더·본문, `Retry-After` 존재 여부와 1·2·3 동시 요청의 제한 재현성
6. 5명×3후보 및 5명×6후보 end-to-end 작업의 진행률·부분 결과·취소·이어받기 UX

### 유료 전환 직후·운영 전

1. 공급자에게 확인한 계약 한도와 실제 제한의 일치 여부
2. 요약정보 단독, 전체정보 단독, 혼합 호출을 각각 1→2→4→8 동시성으로 점진 시험하고 첫 429에서 중단
3. 30건 후보 평가의 p50·p95 완료시간, timeout, 429·5xx 비율
4. 여러 모임이 동시에 계산될 때 전역 큐와 모임별 공정 스케줄링 동작
5. 성공·429·재시도 호출이 사용량 대시보드와 청구 건수에 어떻게 반영되는지 대조
6. 일·월 비용 상한, 모임당 호출 예산, 긴급 차단 스위치 검증
