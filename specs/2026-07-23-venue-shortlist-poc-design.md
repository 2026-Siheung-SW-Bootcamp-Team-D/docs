# 실제 약속 장소 공동 후보 PoC 보완 설계

작성일: 2026-07-23  
대상: `api-validation/poc-v2`  
상태: 설계 승인

## 1. 목적

현재 PoC는 참여자별 대중교통 도달권을 겹치고 교통 거점의 실제 이동시간을 비교해 공정한 만남 지역을 제시한다. 이 기능은 유지한다.

이번 보완은 추천 지역 주변의 음식점·카페·문화시설·관광명소를 검색하고, 사용자가 실제 약속 장소 후보를 함께 축적하고 비교할 수 있도록 두 번째 결정 단계를 추가하는 것이다.

```text
현재 기능
출발지 → 도달권 → 공통 영역 → 교통 거점 → 공정한 지역

추가 기능
공정한 지역 → 실제 장소 탐색 → 공동 후보함 → 이동시간 비교·투표
```

## 2. 설계 원칙

1. 기존 도달권, 교집합, 거점 추천 데이터와 화면을 삭제하거나 대체하지 않는다.
2. 교통 거점은 최종 목적지가 아니라 장소 탐색의 중심점으로 사용한다.
3. 이동 공정성과 사용자의 장소 선호를 별도 지표로 보여준다.
4. 브라우저에는 API 비밀값과 공급자의 원본 응답을 전달하지 않는다.
5. PoC 범위를 유지하고 로그인, 영구 저장, 실시간 다중 접속은 구현하지 않는다.
6. 외부 API 호출량은 공동 후보로 선택된 장소에 대해서만 TMAP을 호출해 제한한다.

## 3. 지도 공급자 결정

### 결정

현재의 Leaflet과 OpenStreetMap 표준 타일을 유지한다. Kakao Map Web SDK는 사용하지 않는다.

### 이유

- Kakao Map Web SDK는 별도의 JavaScript 키와 JavaScript SDK 도메인 등록이 필요하다.
- 현재 환경에서는 도메인 등록이 어렵다.
- Leaflet은 기존 GeoJSON 도달권과 교집합 렌더링을 이미 검증했다.
- 기존 지도를 유지하면 보완 작업을 장소 탐색과 공동 결정 경험에 집중할 수 있다.

### 보완 조건

- 지도 저작자 표기를 `© OpenStreetMap contributors`로 수정한다.
- 표준 OSM 타일은 로컬 PoC와 발표 시연에서만 사용한다.
- 타일을 미리 내려받거나 오프라인 패키지로 만들지 않는다.
- 지도 렌더링을 어댑터 경계로 분리해 향후 다른 공급자로 교체할 수 있게 한다.

## 4. 사용자 흐름

### 4.1 공정한 지역 찾기

기존 흐름을 그대로 사용한다.

1. 참여자 2~5명이 출발지를 검색하고 선택한다.
2. 30분, 45분, 60분 중 도달시간을 선택한다.
3. ODsay 도달권과 전원 교집합을 계산한다.
4. 교집합 안의 교통 거점을 Kakao로 찾는다.
5. TMAP 실제 대중교통 시간으로 공정한 지역 순위를 만든다.

### 4.2 실제 장소 찾기

1. 사용자가 상위 교통 거점 하나를 선택한다.
2. 음식점, 카페, 문화시설, 관광명소 탭 중 하나를 선택한다.
3. 서버가 거점 반경 1km 안의 장소를 Kakao에서 검색한다.
4. 장소 카드와 지도 마커를 함께 표시한다.
5. 사용자는 키워드를 직접 입력해 장소를 추가로 검색할 수 있다.
6. 사용자는 원하는 장소를 공동 후보함에 담는다.

### 4.3 공동 결정

1. 공동 후보는 최대 5개까지 담을 수 있다.
2. 사용자는 후보를 제외하거나 한 표를 줄 수 있다.
3. 사용자가 비교를 실행하면 공동 후보에 대해서만 TMAP 경로를 요청한다.
4. 화면은 이동 공정성 순위와 투표 수를 각각 보여준다.
5. 하나의 숫자로 강제 합산하지 않아 판단 근거가 사라지지 않게 한다.

## 5. 장소 카테고리

PoC의 기본 카테고리는 네 가지다.

| 코드 | 이름 | 검색 방식 |
| --- | --- | --- |
| `FD6` | 음식점 | Kakao 카테고리 검색 |
| `CE7` | 카페 | Kakao 카테고리 검색 |
| `CT1` | 문화시설 | Kakao 카테고리 검색 |
| `AT4` | 관광명소 | Kakao 카테고리 검색 |

세부 취향은 카테고리 코드 대신 Kakao 키워드 검색으로 처리한다.

예: `고기`, `파스타`, `조용한 카페`, `보드게임`, `방탈출`

## 6. 외부 API 흐름

### 6.1 기존 호출

```text
Kakao 출발지 검색
→ ODsay 참여자별 도달권
→ 서버에서 교집합 계산
→ Kakao 교통 거점 검색
→ TMAP 참여자 × 거점 이동시간
```

### 6.2 추가 호출

```text
Kakao 거점 주변 카테고리·키워드 검색
→ 사용자가 공동 후보 선택
→ TMAP 참여자 × 공동 후보 이동시간
```

장소 검색 결과 전체에 TMAP을 호출하지 않는다. 최대 5개의 공동 후보에만 호출한다.

## 7. Kakao 장소 계약

### 7.1 카테고리 검색

```http
GET https://dapi.kakao.com/v2/local/search/category.json
Authorization: KakaoAK {KAKAO_REST_KEY}

category_group_code={FD6|CE7|CT1|AT4}
x={거점 경도}
y={거점 위도}
radius={100..5000}
sort=distance
size=15
```

### 7.2 키워드 검색

기존 `kakaoKeyword()`를 재사용한다.

```http
GET https://dapi.kakao.com/v2/local/search/keyword.json
Authorization: KakaoAK {KAKAO_REST_KEY}

query={사용자 입력}
x={거점 경도}
y={거점 위도}
radius={100..5000}
size=15
```

### 7.3 정규화 결과

브라우저에는 다음 필드만 전달한다.

```text
id
name
category
categoryGroupCode
phone
address
roadAddress
lon
lat
url
distanceMeters
```

## 8. 서버 API

### 8.1 장소 검색

```http
GET /api/venues/search
  ?jobId={작업 ID}
  &hubId={선택한 거점 ID}
  &category={FD6|CE7|CT1|AT4}
  &radius=1000
```

또는:

```http
GET /api/venues/search
  ?jobId={작업 ID}
  &hubId={선택한 거점 ID}
  &query={사용자 키워드}
  &radius=1000
```

서버는 `jobId`와 `hubId`로 기존 작업 결과의 거점 좌표를 조회한다. 브라우저에서 검색 중심 좌표를 직접 받지 않는다.

### 8.2 공동 후보함

```http
GET    /api/jobs/{jobId}/shortlist
POST   /api/jobs/{jobId}/shortlist
DELETE /api/jobs/{jobId}/shortlist/{venueId}
POST   /api/jobs/{jobId}/shortlist/{venueId}/vote
```

PoC 규칙:

- 후보 최대 5개
- 동일 Kakao 장소 ID 중복 금지
- 후보별 투표 수는 0 또는 1
- 서버 재시작 시 데이터 초기화

### 8.3 후보 이동시간 평가

```http
POST /api/jobs/{jobId}/shortlist/evaluate
```

응답은 후보별 참여자 이동시간, 도달 불가 인원, 최장시간, 평균시간, 평균 환승 횟수와 투표 수를 포함한다.

## 9. 화면 구조

기존 지도와 추천 순위 아래에 다음 영역을 추가한다.

```text
이 지역에서 어디로 갈까요?
[음식점] [카페] [문화·놀거리] [관광명소]
[직접 검색 입력] [검색]

장소 카드 목록
장소명 · 세부 카테고리 · 거리 · 주소 · 전화번호
[Kakao 상세보기] [공동 후보에 담기]

공동 후보함
후보 카드 · 투표 · 제외
[실제 이동시간 비교]
```

### 지도 레이어

- 참여자별 도달권
- 전원 공통 영역
- 교통 거점
- 현재 검색된 장소
- 공동 후보

레이어는 토글할 수 있고 교통 거점, 검색 장소, 공동 후보는 서로 다른 마커 스타일을 사용한다.

## 10. 순위와 설명

공동 후보의 이동 공정성 순서는 기존 알고리즘을 재사용한다.

1. 도달 불가 인원 오름차순
2. 최장 이동시간 오름차순
3. 평균 이동시간 오름차순
4. 평균 환승 횟수 오름차순

투표 수는 이동 공정성 순위에 자동 합산하지 않고 별도 배지로 표시한다.

예:

```text
이동 공정성 1위
팀 투표 2표
최장 43분 · 평균 37분 · 도달 불가 0명
```

## 11. 오류 처리

- 존재하지 않는 `jobId` 또는 `hubId`: 404
- 허용하지 않은 카테고리: 400
- 키워드 2자 미만 또는 제한 길이 초과: 400
- 반경 범위 위반: 400
- 후보 5개 초과 또는 중복 후보: 409
- Kakao 장소 검색 실패: 기존 결과 화면은 유지하고 장소 영역에 재시도 안내
- 일부 TMAP 경로 없음: 해당 참여자만 `UNAVAILABLE`로 표시
- 전체 장소 평가 실패: 공동 후보함은 유지하고 다시 비교할 수 있게 함

## 12. 테스트 전략

### 단위 테스트

- Kakao 카테고리 응답 정규화
- 카테고리·키워드·반경 입력 검증
- 공동 후보 추가·중복·제한·제외·투표
- 이동 공정성 순위와 투표 표시 분리

### 통합 테스트

- 장소 검색 API가 작업 결과의 거점 좌표를 사용
- 브라우저가 임의 좌표를 전달할 수 없음
- 후보함 CRUD와 평가 API
- API 키와 인증 헤더가 응답에 포함되지 않음

### 실 API 검증

- `FD6`, `CE7`, `CT1`, `AT4` 각각 한 번 이상 검색
- 사용자 키워드 검색
- 공동 후보 3개에 대한 참여자별 TMAP 비교

### 브라우저 검증

- 기존 출발지부터 교통 거점까지 흐름 유지
- 거점 선택부터 장소 카드 표시까지 동작
- 카드와 지도 마커가 서로 연동
- 후보 추가, 제외, 투표, 비교 동작
- 데스크톱과 390px 모바일 레이아웃
- 브라우저 오류와 경고 0건

## 13. 완료 기준

1. 기존 자동 테스트 32개가 계속 통과한다.
2. 기존 실 API 시나리오 3개의 판정이 유지된다.
3. 상위 교통 거점을 선택하면 네 가지 카테고리의 장소를 검색할 수 있다.
4. 사용자 키워드로 장소를 검색할 수 있다.
5. 장소가 지도와 카드에 함께 표시된다.
6. 공동 후보를 최대 5개까지 추가·제외·투표할 수 있다.
7. 공동 후보에 대해서만 실제 대중교통 이동시간을 비교한다.
8. 외부 API 요청 목적, 메서드, 안전한 URL, 파라미터 이름, 응답시간, 재시도 횟수를 화면에서 확인할 수 있다.
9. API 키 값과 인증 헤더는 브라우저와 보고서에 포함되지 않는다.
10. OpenStreetMap 저작자 표시가 정확하고 Kakao Map 도메인 등록이 필요하지 않다.

## 14. 제외 범위

- Kakao Map Web SDK
- 로그인과 사용자 식별
- 여러 브라우저의 실시간 동기화
- 데이터베이스 영구 저장
- 평점, 리뷰, 영업시간, 메뉴, 가격대 수집
- 운영 트래픽용 지도 타일 공급자 선정
- 운영 환경의 비용·쿼터 최적화

## 15. 남은 제품화 위험

- Kakao의 세부 `category_name`은 안정적인 분류 코드가 아니므로 표시용으로만 사용해야 한다.
- 참여자와 공동 후보가 늘면 TMAP 호출 수가 증가한다.
- OSM 표준 타일은 SLA가 없으므로 운영 서비스에서는 대체 공급자가 필요하다.
- 실제 공동 편집에는 사용자 식별, 동시성 제어와 영구 저장이 필요하다.
