# 후보 장소 보드 중심 제품 재정의

- 작성일: 2026-07-23
- 상태: 기준 설계
- 적용 대상: `docs/specs/`의 현행 명세와 `backend/docs/plan/`
- 대체 대상: 투표·코스·모임 확정·공개 일정·출발 안내를 핵심 흐름으로 삼았던 기존 MVP 설계

## 1. 제품 한 줄 정의

여러 사람이 각자 가고 싶은 장소를 한 보드에 모으고, 지도·좋아요·댓글로 함께 비교한 뒤, 누구나 현재 선택된 장소를 바꿔 보여 주는 약속 장소 결정 보드다.

핵심 가치는 최적의 중간 지점을 자동으로 계산하는 데 있지 않다. 검색 결과를 각자 따로 공유하는 수고를 없애고, 여러 사람의 후보와 반응을 한 화면에 축적해 “지금 어디로 의견이 모였는지”를 빠르게 파악하게 하는 데 있다.

## 2. 문제와 설계 원칙

약속 장소를 정할 때 실제 사용자는 다음 문제를 겪는다.

- 각자가 네이버지도·카카오맵·메신저에서 찾은 장소가 대화 속에 흩어진다.
- 누가 어떤 장소를 좋아하는지, 왜 좋은지 한눈에 보기 어렵다.
- 선택이 바뀔 때마다 이전 맥락을 다시 설명해야 한다.
- 아무도 후보를 떠올리지 못하면 검색을 시작할 지역부터 막힌다.

따라서 제품은 다음 원칙을 따른다.

1. 자동 추천보다 함께 축적하는 경험을 우선한다.
2. 장소는 어느 지도에서 찾았든 좌표와 원본 링크를 보존해 후보로 추가할 수 있어야 한다.
3. 좋아요는 가벼운 선호 표현이며 정식 투표 절차를 만들지 않는다.
4. “확정”이라는 되돌리기 어려운 상태 대신 현재 선택 장소를 보여 주고 언제든 변경·해제한다.
5. 만나기 좋은 지역 추천은 후보가 막혔을 때만 쓰는 보조 수단이다.
6. 공정한 이동 시간보다 사용자가 실제로 가고 싶은 장소를 정하기 쉽게 하는 것을 우선한다.

## 3. MVP 사용자 흐름

1. 한 사람이 약속 보드를 만들고 참여 코드 또는 초대 링크를 공유한다.
2. 참여자는 링크로 입장해 닉네임과 자신의 출발지를 등록한다.
3. 각자는 장소를 검색하거나, 좌표가 확인된 외부 지도 장소 또는 직접 찍은 위치를 후보로 추가한다.
4. 모든 활성 후보는 목록과 지도 핀으로 함께 표시된다.
5. 참여자는 후보에 좋아요를 켜고 끄거나 댓글을 남긴다.
6. 누구나 대화 상황을 반영해 후보 하나를 현재 선택 장소로 표시하거나 선택을 해제한다.
7. 후보를 떠올리기 어려울 때 누구나 출발지와 허용 시간을 기준으로 만나기 좋은 지역을 요청한다.
8. 참여자는 제안된 지역 주변을 카테고리 또는 키워드로 탐색하고 발견한 장소를 기존 후보 보드에 추가한다.

지역 제안 이후에도 별도의 추천 결과나 코스가 만들어지지 않는다. 최종 산출물은 계속 같은 후보 장소 보드이며, 지역 탐색은 후보를 더 쉽게 추가하기 위한 진입점이다.

## 4. 핵심 개념

### 4.1 보드

- 상태는 `OPEN`, `CLOSED`만 사용한다.
- `CLOSED`는 오래된 보드를 관리하기 위한 보관 상태일 뿐, 모임 확정을 의미하지 않는다.
- MVP 화면에서는 닫기 기능을 노출하지 않아도 된다.
- 보드는 `selectedPlaceId`를 0개 또는 1개 가진다.
- 보드를 만든 참여자는 생성자로 기록할 수 있지만 다른 참여자보다 더 많은 기능 권한을 갖지 않는다.
- 모든 참여자는 보드 화면에서 참여 코드와 초대 링크를 언제든 다시 확인하고 복사할 수 있다.

### 4.2 후보 장소와 현재 선택 장소

- 보드에 추가된 활성 장소는 모두 후보 장소다.
- 현재 선택 장소는 별도 종류의 장소가 아니라 보드가 가리키는 후보 하나다.
- `selectedPlaceId`가 없으면 아직 의견이 모이지 않은 상태다.
- 모든 참여자는 현재 선택 장소를 다른 후보로 변경하거나 해제할 수 있다.
- 마지막으로 성공한 변경을 현재 상태로 사용하고, 변경한 참여자와 시각을 함께 표시한다.
- 선택된 장소를 삭제하면 선택 포인터를 먼저 해제한 뒤 장소를 보관 처리한다.
- 이 구조로 한 보드에 여러 “선택됨” 상태가 생기는 문제를 막는다.

### 4.3 좋아요

- 좋아요는 참여자 한 명이 장소 하나에 최대 하나만 남기며, 서로 다른 여러 장소에는 각각 남길 수 있다.
- 켜기와 끄기는 멱등적이어야 한다.
- 집계 수와 내 좋아요 여부를 장소 조회에 함께 제공한다.
- 투표 생성·마감·복수 선택 같은 절차는 두지 않는다.

### 4.4 댓글

- 모든 참여자는 후보 장소에 의견을 남길 수 있다.
- 댓글 삭제는 작성자에게만 허용한다.
- 삭제된 댓글은 목록에서 제외하되 감사와 복구 가능성을 위해 소프트 삭제할 수 있다.

### 4.5 출발지

- 출발지는 지역 제안 계산에만 사용한다.
- 다른 참여자에게 상세 주소를 노출하지 않는다.
- 저장이 필요하면 암호화하며, 응답에는 설정 여부와 표시용 요약만 반환한다.

## 5. 장소 유입 정책

### 5.1 공식 검색

MVP의 기본 검색 공급자는 현재 연동된 카카오 로컬 검색이다. 검색 결과는 아직 보드의 후보가 아니며, 사용자가 명시적으로 추가할 때만 장소 레코드를 만든다.

검색 API와 저장 모델은 공급자 중립적으로 설계한다.

- `sourceProvider`: `KAKAO`, `NAVER`, `EXTERNAL`, `MANUAL`
- `providerPlaceId`: 공급자가 제공할 때만 저장
- `sourceUrl`: 상세 확인을 위한 원본 지도 링크
- 장소명, 주소, 경도, 위도, 카테고리

네이버 공식 검색 또는 링크 해석은 별도 PoC를 통과한 뒤 어댑터로 추가한다. 네이버 검색이 없다는 이유로 후보 보드나 지역 제안 개발을 막지 않는다.

### 5.2 외부 지도 링크

원본 링크는 사용자가 상세 정보와 리뷰를 해당 지도에서 확인하도록 보존한다. 서비스가 외부 HTML을 크롤링하거나 리뷰·별점을 복제하는 기능은 보장하지 않는다.

- 좌표와 장소명이 공식 검색 선택 또는 사용자 입력으로 확인된 경우에만 후보를 생성한다.
- 좌표를 알 수 없는 링크만으로 장소를 생성하지 않는다.
- 허용된 지도 호스트와 `https` 링크만 저장하고, 서버가 임의 리다이렉트를 따라가며 메타데이터를 수집하지 않는다.

### 5.3 직접 위치 지정

검색으로 찾지 못한 장소는 지도 핀과 이름을 직접 지정해 추가할 수 있다. 좌표는 대한민국 서비스 범위와 유효한 경위도인지 검증한다.

## 6. 만나기 좋은 지역 제안

지역 제안은 “정답 장소 추천”이 아니라 탐색 범위를 좁히는 fallback이다.

### 6.1 입력

- 모든 활성 참여자의 출발 좌표
- 허용 이동 시간: 30분, 45분, 60분
- 계산 시작 시점의 참여자와 출발지 스냅샷

### 6.2 처리 파이프라인

1. 참여자별 ODsay 대중교통 도달권을 조회한다.
2. JTS로 도달권 다각형의 공통 교집합을 계산한다.
3. 의미 있는 면적의 조각을 큰 순서로 추린다.
4. 각 조각 내부에서 카카오 장소 검색으로 역·상권 등 탐색 기준점을 수집한다.
5. 중복과 조각 외부 결과를 제거하고 최대 1~3개의 지역 제안을 반환한다.

지역 제안 순서는 화면 배치를 위한 안정적인 순서일 뿐, 이동 시간 공정성 점수가 아니다. 공통 영역이 없으면 시간을 늘려 다시 시도하도록 안내한다. 서울–부산 같은 장거리 모임의 해결은 MVP 범위에서 제외한다.

### 6.3 비용 경계

- 지역 계산 중 TMAP 경로 API를 모든 참여자×후보 조합으로 호출하지 않는다.
- ODsay와 카카오 응답은 입력 좌표, 시간, 검색 조건을 정규화한 키로 캐시한다.
- 중복 실행을 막고 작업 상태를 재사용한다.
- 실제 경로 시간은 사용자가 특정 후보를 눌렀을 때만 대표 경로를 온디맨드로 조회하는 후속 기능으로 둔다.

## 7. 권한

보드 생성자는 개설 이력을 나타낼 뿐 관리자 역할이 아니다. 보드에 참여한 사람은 모두 같은 기능 권한을 가진다.

| 작업 | 모든 참여자 |
|---|---:|
| 참여 코드·초대 링크 확인·공유 | 가능 |
| 보드 정보 변경 | 가능 |
| 자신의 출발지 등록·수정 | 가능 |
| 후보 장소 추가·삭제 | 가능 |
| 좋아요 추가·취소 | 가능 |
| 댓글 작성 | 가능 |
| 자신의 댓글 삭제 | 가능 |
| 현재 선택 장소 지정·변경·해제 | 가능 |
| 지역 제안 실행 | 가능 |

보드 폐쇄·삭제처럼 전체 사용을 중단시키는 관리 기능은 MVP 화면에서 제공하지 않는다. 현재 선택 장소는 모든 참여자가 바꿀 수 있고 마지막으로 성공한 요청이 현재 상태가 된다.

## 8. API 계약 방향

상세 스키마는 API 명세서에서 정의하되 다음 자원 경계를 사용한다.

- 보드·초대·참여자
- 장소 검색 후보
- 보드 후보 장소
- 장소 좋아요
- 장소 댓글
- 현재 선택 장소
- 지역 제안 작업

핵심 명령은 다음과 같다.

- `POST /boards/{boardId}/places`: 검색 결과, 외부 출처 또는 직접 위치를 후보로 추가
- `GET /boards/{boardId}/places`: 후보 목록, 좋아요 수, 내 좋아요, 현재 선택 여부 조회
- `PUT /boards/{boardId}/places/{placeId}/likes/me`: 좋아요
- `DELETE /boards/{boardId}/places/{placeId}/likes/me`: 좋아요 취소
- `GET /boards/{boardId}/invitation`: 모든 참여자가 참여 코드와 초대 링크 조회
- `PUT /boards/{boardId}/selected-place`: 참여자가 현재 선택 장소 지정 또는 변경
- `DELETE /boards/{boardId}/selected-place`: 참여자가 선택 해제
- `POST /boards/{boardId}/area-search-jobs`: 참여자가 지역 제안 작업 시작
- `GET /boards/{boardId}/area-search-jobs/{jobId}`: 작업 상태와 최대 1~3개 지역 조회

투표, 코스, 모임 확정, 공개 일정, 출발 알림 API는 새 MVP 계약에서 제거한다.

## 9. 데이터 모델 방향

핵심 테이블은 다음으로 단순화한다.

- `board`: 보드 상태, nullable `selected_place_id`, 마지막 선택 변경 참여자와 시각
- `participant`: 역할, 토큰, 출발지
- `place`: 후보 장소와 공급자·원본 링크·좌표
- `place_like`: 참여자별 장소 좋아요, `(place_id, participant_id)` 유일
- `place_comment`: 장소 댓글
- `area_search_job`: 비동기 지역 계산 작업과 입력 스냅샷
- `area_suggestion`: 공통 영역 안의 지역 탐색 기준점

기존 `vote`, `vote_option`, `vote_ballot`, `course_draft`, `course`, `course_stop`, `departure_calculation`은 새 핵심 모델에서 제거한다. 개발 환경의 기존 데이터 호환성보다 새 계약의 단순성과 일관성을 우선한다.

## 10. 오류와 동시성

- 보드가 열려 있지 않거나 참여자 토큰이 유효하지 않으면 변경을 거부한다.
- 다른 보드의 장소를 현재 선택 장소로 지정할 수 없다.
- 보관된 장소에는 좋아요·댓글·선택 변경을 허용하지 않는다.
- 선택 변경은 last-write-wins로 처리하고 응답에 변경 참여자와 시각을 포함한다.
- 같은 좋아요 요청을 반복해도 결과와 집계가 흔들리지 않아야 한다.
- 동일한 지역 검색 입력의 실행 중 작업 또는 유효한 완료 결과가 있으면 재사용한다.
- 외부 API 일부 실패는 공급자와 단계가 드러나는 오류로 반환하되 비밀 키와 원문 민감 정보는 노출하지 않는다.

## 11. 보안과 개인정보

- 참여자 토큰은 원문을 저장하지 않고 해시로 검증한다.
- 출발지 상세 정보는 최소 수집하고 저장 시 암호화한다.
- 외부 API 키는 서버 환경변수 또는 비밀 저장소에서만 주입한다.
- 외부 링크는 프로토콜과 호스트를 검증해 `javascript:` URL과 임의 내부 주소 접근을 막는다.
- 검색어와 댓글 길이를 제한하고, API별 속도 제한을 적용한다.
- 보드 응답은 다른 참여자의 상세 출발 주소나 토큰을 포함하지 않는다.

## 12. MVP 완료 기준

다음 흐름을 실제 브라우저에서 처음부터 끝까지 수행할 수 있어야 한다.

1. 보드 생성, 초대, 두 명 이상 참여
2. 각 참여자의 출발지 등록
3. 카카오 검색 결과와 직접 지정 위치를 후보로 추가
4. 후보가 목록과 지도 핀에 동시에 표시
5. 서로 다른 참여자의 좋아요와 댓글 반영
6. 서로 다른 참여자의 현재 선택 장소 지정, 변경, 해제와 마지막 변경자 표시
7. 선택된 후보 삭제 시 선택 자동 해제
8. 임의 참여자가 후보가 막혔을 때 지역 제안 실행
9. 제안 지역 주변을 키워드·카테고리로 탐색해 후보 추가

지역 제안 검증에서는 ODsay 호출, 교집합 계산, 카카오 기준점 탐색을 확인하고 TMAP 호출이 없음을 확인한다. 장거리 공통 영역, 네이버 링크 자동 해석, 외부 별점·리뷰 복제, 코스·일정 확정은 완료 기준에 포함하지 않는다.

## 13. 문서와 구현 전환

1. `docs/specs/`의 기능 요약, 기능 명세, API, ERD, 시스템 아키텍처를 이 설계에 맞춘다.
2. 과거 방향을 설명하는 PoC 설계는 `specs/archive/`로 이동해 역사 자료로만 남긴다.
3. `backend/docs/plan/`은 후보 장소 보드 중심으로 단계 이름과 의존성을 다시 정리한다.
4. 이미 구현된 투표·코스·출발 기능은 새 계약에 맞춰 제거하거나 비활성화하는 breaking migration 계획을 세운다.
5. 기존 Phase 6은 지역 제안 fallback 구현 단계로 축소하고, TMAP 전수 경로 평가와 공정성 순위를 제거한다.
