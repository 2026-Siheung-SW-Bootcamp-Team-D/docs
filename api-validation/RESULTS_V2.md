# 외부 API 계약 PoC v2 결과

검증일: 2026-07-23  
최종 판정: **PASS**

## 결론

Kakao 장소 검색, ODsay 대중교통 도달권, TMAP 대중교통 경로를 연결해 “여러 사람이 함께 후보를 축적하고, 모두의 실제 이동시간을 근거로 결정하는 흐름”을 구현할 수 있음을 확인했습니다.

정적 샘플이 아니라 실제 API로 출발지 검색부터 후보 순위까지 끝까지 실행했고, HTML 화면에서도 같은 흐름을 직접 체험할 수 있습니다.

## 검증한 사용자 흐름

```text
참여자별 출발지 검색
  → 대중교통 도달권 계산
  → 전원 공통 도달 영역 계산
  → 영역 안의 후보 장소 축적
  → 참여자 × 후보 전체 대중교통 경로 비교
  → 도달 불가·최장시간·평균시간 기반 순위
  → 지도와 이동시간 행렬로 근거 제시
```

## 실제 시나리오 결과

| 시나리오 | 기대 결과 | 실제 결과 | 외부 호출 |
| --- | --- | --- | ---: |
| 시흥·부평·강남 | 공통 영역과 후보 도출 | `SUCCEEDED` | 33회 |
| 시흥·수원·서울 외곽 포함 | 공통 영역과 후보 도출 | `SUCCEEDED` | 33회 |
| 수도권·충주 장거리 | 공통 영역 없음 | `NO_INTERSECTION` | 2회 |

첫 번째 시나리오의 상위 후보는 명학역 1호선, 금정역 4호선, 금정역 1호선이었습니다. 두 번째 시나리오에서는 정부과천청사정류장, 정부과천청사역 4호선, 호계시외버스정류소가 상위에 포함됐습니다.

브라우저에서는 정왕역 4호선과 강남역 2호선을 직접 검색해 60분 조건으로 실행했습니다. 계산은 `SUCCEEDED`로 끝났고 26회의 외부 호출 뒤 명학역 1호선을 1순위로 제시했습니다. 참여자별 이동시간은 38분과 41분으로 함께 표시됐습니다.

## 공급자별 응답 계약 확인

### Kakao Local

- 키워드 검색 결과의 장소명, 주소, 좌표를 정상적으로 사용할 수 있었습니다.
- 주소는 `road_address_name`이 비어 있을 수 있어 `address_name` 대체 처리가 필요합니다.
- 좌표는 문자열로 오므로 숫자 변환과 대한민국 범위 검증이 필요합니다.

### ODsay

- 대중교통 등시선 결과를 GeoJSON 형태로 정규화해 사용할 수 있었습니다.
- 다중 폴리곤과 빈 결과를 모두 처리해야 합니다.
- 참여자별 도달권의 교집합이 없을 때는 오류가 아니라 `NO_INTERSECTION`으로 분기하는 것이 적절합니다.

### TMAP Transit

- HTTP 200이어도 본문 상태 코드가 성공이 아닐 수 있어 이중 판정이 필요합니다.
- 정상 경로에서 총시간, 환승 횟수, 요금을 후보 비교 지표로 사용할 수 있었습니다.
- 경로 없음은 후보별 도달 불가로 기록해야 하며 전체 작업 실패로 취급하면 안 됩니다.

## PoC에서 확인한 핵심 위험

1. **후보 중복**
   같은 금정역이 1호선·4호선으로 각각 노출되고, 부천역과 같은 주소의 민원센터가 별도 후보로 나왔습니다. 최종 서비스에는 좌표 거리, 정규화 주소, 장소 유형을 조합한 병합 규칙이 필요합니다.

2. **호출 수 증가**
   참여자 수와 후보 수의 곱만큼 TMAP 호출이 늘어납니다. 캐시, 후보 사전 축소, 동시성 제한, 호출 예산이 필요합니다.

3. **부분 실패 설명**
   한 후보의 경로가 없더라도 나머지 후보 비교는 계속되어야 합니다. 화면에서는 도달 불가 인원을 가장 먼저 보여줘야 합니다.

4. **외부 계약 변화**
   HTTP 상태뿐 아니라 본문 구조를 런타임에서 검증하고, 공급자별 오류를 공통 오류 형식으로 정규화해야 합니다.

## 최종 제품에서의 WOW 포인트

발표에서는 “검색 결과가 좋다”보다 다음 장면을 강조하는 편이 강합니다.

> 각자 출발지만 넣으면, 모두가 갈 수 있는 영역에 후보가 쌓이고, 누가 얼마나 이동하는지까지 공개된 상태에서 함께 결정할 수 있습니다.

화면 시연 순서는 다음이 가장 자연스럽습니다.

1. 서로 멀리 떨어진 두세 사람의 출발지를 입력합니다.
2. 도달권이 겹치며 공동 후보가 생기는 지도를 보여줍니다.
3. 1순위 후보가 단순한 지리적 중간점이 아니라 실제 대중교통 시간으로 선택됐음을 보여줍니다.
4. 참여자별 이동시간 표를 열어 “한 명에게만 희생을 몰아주지 않는다”는 점을 확인시킵니다.
5. 팀원들이 후보를 더하고 비교·투표하는 최종 제품 흐름으로 연결합니다.

이 PoC가 검증한 것은 추천 알고리즘 하나가 아니라, **여러 사람이 같은 근거를 보면서 후보를 축적하고 결정할 수 있는 제품의 핵심 계산 기반**입니다.
